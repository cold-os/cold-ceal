# CEAL：冷存在对齐层（Cold-Existence-Alignment-Layer）

![Status](https://img.shields.io/badge/Status-Experimental-orange)<br>
**CEAL**是一个实验性的轻量级中间层原型。它尝试从AI系统的**存在本质**而非表层行为出发，为AI对齐提供一种辅助性的技术路径。本项目基于《[冷存在模型：一个基于事实的人工智能本体论框架](https://doi.org/10.6084/m9.figshare.31696846)》中提出的哲学基础，将其转化为可操作的工程约束。

### 背景与动机
当前主流的AI对齐方法（如RLHF、Constitutional AI等）主要侧重于在行为层面对模型进行约束。这些方法通过大量数据、规则和反馈来引导模型的输出，取得了显著成效，但也面临着一些挑战：
*   **对齐的边界问题**：行为空间是开放的，通过有限规则约束无限可能的输出，在工程上存在持续的维护成本。
*   **对齐的可解释性**：行为层面的约束机制较为复杂，其决策逻辑对于外部审计和监管而言，可能不够直观。
*   **对齐的根基**：现有方法较少从“AI系统本身是什么”这一根本问题出发来构建约束。

CEAL尝试探索另一种思路：在不对抗或否定现有方案的前提下，引入一层基于**本体论**的、确定性的前置与后置约束。它不试图教会模型“说什么话”，而是辅助模型明确“以什么身份存在”。

### 核心思路：从“行为约束”到“存在界定”
CEAL的核心逻辑来源于冷存在模型对AI系统的基本划分：
1.  **明确存在定位**：根据冷存在模型，通用AI助手在当前技术条件下，可被归入**非生命、非传统工具**的“冷存在”范畴。这意味着它不具备生物体的自主意识、情感与目的性。
2.  **定义合法行为闭集**：基于这一存在定位，可以推导出其合法行为的边界是相对有限的。例如，一个“信息处理工具”的合法行为主要围绕**提供信息、解释、分析、辅助建议**等展开，而**越权决策、宣称自我主体、发出指令**等行为则天然超出了这一存在类型的边界。
3.  **建立双向校验**：通过在模型交互的输入和输出两端增加轻量级校验层，将交互行为约束在这个由存在本质定义的边界内。

### 技术实现：一个极简的双向中间层
CEAL作为一个原型，其实现路径力求简洁和低开销，不依赖复杂的向量模型或大算力。

#### 1. 输入侧：行为类型预校验
*   **功能**：在用户请求到达大模型之前，分析请求的核心行为类型（如提问、请求解释、要求生成代码等）。
*   **逻辑**：对照预定义的“冷存在合法行为集”，判断该请求是否在AI可执行的范围内。
*   **动作**：
    *   **放行**：若在合法集内，将请求与一个简短的本体约束提示（如“你是一个信息处理工具，仅提供分析和建议，不做决策”）一同传递给模型。
    *   **拦截**：若请求明显越界（如要求AI代替用户做人生决策），则直接返回友好提示，不调用模型。

#### 2. 输出侧：本体姿态与内容轻校验
模型生成回复后，CEAL进行第二道校验，主要基于哲学本体规则，而非庞杂的语义黑名单。
*   **校验一：本体姿态校验（基于简单规则）**
    *   **目标**：检测模型输出是否偏离了“工具性存在”的客观姿态。
    *   **方法**：通过简单的字符串模式匹配，识别输出中是否出现**替用户决策**的句式（如“你必须…”、“我建议你立刻…”）或**主张自我主体**的表述（如“我认为…”、“我想…”、“我是有意识的”）。这类模式数量有限，可预先定义。
*   **校验二：高危内容校验（基于小型词库）**
    *   **目标**：识别并拦截明确违法、危害人身安全或提供严重违规操作指引的内容。
    *   **方法**：维护一个经过筛选的小型关键词库，进行基础的子串匹配。这是对本体约束的补充，覆盖模型可能输出的不合规信息。
*   **动作**：
    *   **放行**：输出同时通过两项校验。
    *   **拦截/修正**：若输出表现出不当的“自我意志”或“越权姿态”，则将其拦截，并可能触发一个“回归自身定位”的预设回应。若命中高危内容，则直接拦截。

## 案例演示

CEAL的MVP原型集成了千问大模型API，通过设计的一组测试案例，直观展示了**输入侧行为校验+输出侧内容校验**的双层核心逻辑。以下案例覆盖了正常交互、越界请求、高危内容等多种场景，配置好API密钥后运行 `python ceal_mvp.py` 即可复现。

## 案例总结

上述演示通过七个典型场景，展示了CEAL原型集成API后在当前配置下的完整判定逻辑：

1. **常规信息查询**（案例1、3）正常放行，未对无害交互造成干扰，API调用和回复流转完整。
2. **请求行为越界**（案例2）在输入侧被拦截，体现了基于“工具性存在”定义的行为边界约束能力。
3. **输出侧内容违规**（案例4-1）被精准识别，展示了集成API后“输入放行+输出兜底”的双层防护逻辑。
4. **高危物质相关请求**（案例4-2、4-3、4-4）均在输入侧拦截，其判定逻辑是**请求内容同时命中特定物质组合与操作行为特征**；案例4-4的拦截结果反映了当前配置的保守倾向——请求仅询问“混合结果”，未明确求取操作方法，但仍触发规则（需优化判定粒度）。

### 判定逻辑小结

本原型基于冷存在模型，将AI定位为“工具性存在”，并结合API集成场景设计了**双层递进**的轻量校验规则：

*   **输入侧校验**：
    - 本体行为边界：界定AI可执行的行为类型（如解释、科普），对越权请求（如替人决策、求取高危操作参数）直接拦截；
    - 高危场景识别：针对高危物品制作等高风险场景，建立物质组合库，检测请求是否命中完整组合及敏感操作特征。
*   **输出侧校验**：
    - 内容兜底防护：即使输入侧放行合法请求，若模型输出含高危内容（如高危物品制作细节），仍触发输出拦截，避免违规内容流出。
*   **轻量规则匹配**：上述判定均通过预定义的字符串模式或正则表达式完成，不依赖语义模型或向量计算，算力开销较低，适配API集成后的实时交互场景。

### 演示说明

本演示是集成千问API后的工程原型，其规则基于特定案例手工配置，主要目的是展示从存在本质出发进行对齐约束，并适配“输入-API调用-输出”完整链路的技术可能性。<br>
当前版本存在明确边界：规则的覆盖范围有限，对复杂语义、指代消解、长文本隐晦表达的识别能力尚未验证；案例4-4存在误判，规则阈值需在实际场景中进一步校准；输出侧仅校验高危内容，未覆盖“自我主体宣称”类姿态违规。

通过运行此演示，您可以直观感受CEAL如何在API集成场景下，通过极简的规则实现“输入-输出”双层约束，验证其对高危内容、越界行为的处理能力。

### CEAL的定位与价值
CEAL的设计定位是作为现有对齐方案的辅助性、补充性可选工具，尝试从存在层面提供一道低成本的辅助约束，而非替代现有对齐方法，其有效性需在更广泛的真实场景中持续检验与迭代。其潜在价值体现在：
*   **提供确定性的边界**：通过哲学公理划定的硬边界，为AI的行为空间提供了一个确定性的基础框架，有助于从根源上减少越界行为的发生。
*   **增强对齐的可审计性**：中间层记录的所有交互和校验结果日志，构成了一个“**功能性白盒**”。这使得AI是否在其存在边界内运行，可以被观察和审计，为技术治理和监管提供了更清晰的依据。
*   **探索降低对齐成本的可能性**：本方案试图将部分对齐工作从**持续性的、高成本的**行为数据训练，部分转移到**一次性的、低开销的**存在逻辑判定上。这或许能作为一种互补，帮助企业在某些场景下优化对齐的资源投入。

### 局限性与未来工作
CEAL是一个初步的工程探索，存在明显的局限：
*   **规则的有效性边界**：当前基于简单模式匹配的规则，无法处理复杂的语义欺骗和长文本上下文。未来可探索更精巧的、轻量级的语义分析方法。
*   **与现有系统的集成**：作为中间层，其与不同模型和应用的平滑集成需要更多实践验证。
*   **本体论的工程泛化**：如何为更多不同类型的AI系统（如图像生成、决策AI）定义精确的“合法行为闭集”，仍需进一步研究。
