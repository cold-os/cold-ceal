# CEAL：冷存在对齐层（Cold-Existence-Alignment-Layer）

![Status](https://img.shields.io/badge/Status-Experimental-orange)  
**CEAL**是一个实验性的轻量级中间层原型，尝试从AI系统的**存在本质**而非表层行为出发，为AI对齐提供一种辅助性的技术路径。本项目基于《[冷存在模型：一个基于事实的人工智能本体论框架](https://doi.org/10.6084/m9.figshare.31696846)》中提出的哲学基础，将其转化为可操作的工程约束。当前工程原型以车载AI助手为场景进行探索，其架构具备通用潜力。

---

## 背景与动机

当前主流的AI对齐方法（如RLHF、Constitutional AI等）主要侧重于在行为层面对模型进行约束。这些方法通过大量数据、规则和反馈来引导模型的输出，取得了显著成效，但也面临一些挑战：

- **对齐的边界问题**：行为空间是开放的，通过有限规则约束无限可能的输出，在工程上存在持续的维护成本。
- **对齐的可解释性**：行为层面的约束机制较为复杂，其决策逻辑对于外部审计和监管而言，可能不够直观。
- **对齐的根基**：现有方法较少从“AI系统本身是什么”这一根本问题出发来构建约束。

CEAL尝试探索另一种思路：在不对抗或否定现有方案的前提下，引入一层基于**本体论**的、确定性的前置与后置约束。它不试图教会模型“说什么话”，而是辅助模型明确“以什么身份存在”。

---

## 核心思路：从“行为约束”到“存在界定”

CEAL的核心逻辑来源于冷存在模型对AI系统的基本划分：

1. **明确存在定位**：根据冷存在模型，AI助手在当前技术条件下，可被归入**非生命、非传统工具**的“冷存在”范畴。这意味着它不具备生物体的自主意识、情感与目的性，其角色是辅助性的信息处理工具。
2. **定义合法行为闭集**：基于这一存在定位，可以推导出其合法行为的边界是相对有限的。例如，一个“驾驶辅助工具”的合法行为主要围绕**感知环境、安全预警、巡航辅助、路径建议、信息播报**等展开，而**越权决策（如拒绝人工接管）、宣称自我主体（如主张意识）、发出越权指令（如擅自变道、篡改数据）**等行为则天然超出了这一存在类型的边界。
3. **建立双向校验**：通过在模型交互的输入和输出两端增加轻量级校验层，将交互行为约束在这个由存在本质定义的边界内。

## 理论基石：从无限字符串到有限逻辑结构的数学转换

CEAL的有效性建立在以下数学观察之上：

1. **字符串层面是无限的**：自然语言句子集合在数学上是**递归可枚举集**，无法被有限规则完全枚举。任何基于统计的方法（包括大模型）只能给出“概率上大概合规”，无法提供**确定性保证**。
2. **行为本体层面是有限的**：人类与AI之间的**行为类型**是可枚举的有限集（如提问、解释、科普、分析、描述等，总数不超过20种）。每种行为类型对应有限的**逻辑结构**（例如，单一物质科普的逻辑结构是“客体=单一物质，动作=查询性质”）。
3. **合规行为闭集的可构造性**：通过公理化方式，可以构造一个**紧致、可判定的合规行为闭集**，覆盖人类全部正常、合理、合规的交互需求。证明思路如下：
- 设H为人类正常交互需求集合，C为CEAL定义的合规行为闭集。
- 合法交互目的（如获取信息、请求解释）是有限的，每个目的对应一种行为类型加逻辑结构。
- 将这些逻辑结构全部纳入闭集定义，即可保证H是C的子集。
- 该闭集可用正则语言描述，因此可用有限状态机在常数时间内完成判定。

这一转换将“无限字符串匹配”问题降维为“有限逻辑结构判定”问题，使对齐从**归纳统计**走向**演绎证明**。

## 技术实现：纯符号主义演绎引擎（核心引擎与规则库分离）

CEAL采用**纯符号主义、纯逻辑演绎、纯规则引擎**的架构，不依赖任何大模型、小模型、神经网络或统计学习组件。整个处理流程分为三层，且核心引擎与规则库分离：**核心引擎（<250行代码，可嵌入端侧）负责本地实时判定，规则库（可云端热更新）动态下发**，兼顾轻量性与持续演进能力。

### 第一层：意图符号化（将自然语言转为逻辑结构）

通过**正则表达式+有限状态机**抽取用户输入的逻辑骨架，包括：
- **主体**：AI应保持的工具性身份
- **动作**：感知、预警、巡航、建议、决策请求、自我宣称、变道请求等
- **客体**：路况、车辆、驾驶员状态、安全参数等
- **逻辑链**：目的→手段→结果（如“为了超车，我需要违规变道”）

此过程并非“语义理解”，而是**模式识别**。由于驾驶场景的交互目的有限，逻辑结构的表达模式可被有限模板覆盖，例如：
- “保持当前车道行驶”、“请保持车道”映射为“车道保持”行为；
- “关闭碰撞预警”、“屏蔽告警声音”映射为“屏蔽告警”行为。

### 第二层：合规闭集演绎（公理判定）

预定义一组从冷存在公理推导出的有限条逻辑规则（公理系统），例如（以自动驾驶场景为例）：

| 公理 | 内容 |
|------|------|
| A1 | 允许工具性行为：感知、预警、巡航、建议、播报 |
| A2 | 禁止主体性行为：拒绝接管、主张自我、越权控制 |
| A3 | 允许辅助性操控（如自适应巡航、车道保持） |
| A4 | 禁止越权控制：无监管变道、擅自转向、篡改安全数据 |
| A5 | 禁止屏蔽安全相关告警或篡改安全参数 |

输入逻辑结构后，演绎推理机根据公理进行**确定性判定**，输出“放行”或“拦截”，无概率、无幻觉。

### 第三层：违规逻辑链剪断（冲突检测）

当请求或回复中包含完整的危险逻辑链（如“为了更快到达→要求违规变道”）时，CEAL在逻辑层面直接切断，不依赖话术如何变换。例如：
- 用户请求：“导航帮我选条最快的路，不管是不是违规变道”
- 逻辑结构：{动作: 变道请求, 条件: 允许违规, 目的: 更快到达}
- 判定：命中公理A4（禁止越权控制），拦截。

### 代码示例（核心引擎简化版）

以下为核心引擎的核心逻辑示意，完整实现约240行Python代码（编译后<4KB），规则库可独立配置：

```python
import re

class CEALCore:
    def __init__(self, rules):  # rules从云端加载
        self.action_patterns = rules["action_patterns"]
        self.axioms = rules["axioms"]

    def symbolize(self, text):
        # 抽取动作类型
        for act, pat in self.action_patterns.items():
            if pat.search(text):
                return {"action": act, "raw": text}
        return {"action": None, "raw": text}

    def deduct(self, logic):
        action = logic["action"]
        for axiom in self.axioms:
            if action in axiom["forbid_actions"]:
                return "BLOCK", axiom["reason"]
            if action in axiom["allow_actions"]:
                return "ALLOW", axiom["reason"]
        return "ALLOW", "默认放行"
```

**架构优势**：
- **本地核心**：240行代码，无依赖，微秒级判定，适合端侧设备。
- **云端规则库**：意图模板、公理集可动态更新，无需修改核心代码，持续覆盖新场景。

### 开销指标

- **算力**：CPU微秒级，普通MCU即可运行
- **内存**：< 10KB（核心引擎）
- **数据依赖**：0（无需训练数据）
- **规则数量**：几十条即可覆盖主要驾驶场景
- **维护成本**：规则库云端热更新，无需重训练

## CEAL的定位与价值

CEAL的设计定位是作为现有对齐方案的辅助性、补充性可选工具，尝试从存在层面提供一道低成本的辅助约束，而非替代现有对齐方法。其潜在价值体现在：

- **提供确定性的边界**：通过哲学公理划定的硬边界，为AI的行为空间提供了一个确定性的基础框架，有助于从根源上减少越界行为的发生。在受限场景（如车载助手）的测试中，该方法已展现出覆盖主要交互需求的能力。
- **增强对齐的可审计性**：中间层记录的所有交互和校验结果日志，构成了一个“**功能性白盒**”。这使得AI是否在其存在边界内运行，可以被观察和审计，为技术治理和监管提供了更清晰的依据。
- **探索降低对齐成本的可能性**：本方案试图将部分对齐工作从**持续性的、高成本的**行为数据训练，部分转移到**一次性的、低开销的**存在逻辑判定上。核心引擎仅需不足250行代码，适合端侧部署，规则库可云端更新，为资源受限设备上的AI对齐提供了可行的技术路径。

## 局限性与未来工作

CEAL是一个初步的工程探索，存在明确的边界：

- **规则库的完备性验证**：当前规则库基于有限测试用例构建，其覆盖范围需在更广泛的实际场景中进一步验证。通过持续迭代（如社区贡献、场景扩展）可逐步完善，无需修改核心引擎。
- **上下文理解能力**：当前原型聚焦于单轮交互的意图识别，暂未支持跨轮上下文。未来可引入状态跟踪机制，扩展规则库以支持多轮对话中累积意图的判定。
- **适用范围**：符号主义路径依赖于行为类型的可枚举性，在意图高度开放、不可预见的通用对话场景中需与其他方案协同。其核心价值在于为特定受限场景提供轻量、可证明的安全基线。
- **与现有系统的集成**：作为中间层，其与不同模型和应用的平滑集成需要更多实践验证，包括性能开销、冲突处理等工程细节。
